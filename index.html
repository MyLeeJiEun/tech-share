第一层 · DTK

DPasswordEdit（DTK 控件）初始化时已显式关闭输入法

lineEdit()->setEchoMode(QLineEdit::Password);
lineEdit()->setAttribute(Qt::WA_InputMethodEnabled, false);

切换明文只改 EchoMode，没有重新打开输入法，也没有重新设置 WA_InputMethodEnabled。

问题：设置了 WA_InputMethodEnabled=false 的 QLineEdit，为什么从 Password 切到 Normal 后输入法又恢复了？

我最初的反应是：是不是 DTK 切换 EchoMode 时漏掉了什么？于是手动在 setEchoMode(Normal) 之后补了一行 setAttribute(Qt::WA_InputMethodEnabled, false) —— 完全没有用。这个现象第一次告诉我：问题可能不在 DTK 层，而是 Qt ，当然此时仅仅是我的猜测。

Part 1 · 链路
07 / 23
两个关键事件：InputMethodQuery vs InputMethod

切到 Normal 后发生了什么

InputMethodQuery
输入法 → 查询控件
问"能接收输入法吗？"
InputMethod
输入法 → 提交控件
提交候选词/预编辑
这次问题卡在查询阶段。
setEchoMode(Normal)
→ setInputMethodHints() 清除了 ImhHiddenText / ImhSensitiveData
→ QInputMethod::update(Qt::ImHints)
→ IBus 发现不再是 HiddenText → 重新查询 ImEnabled
→ QWidget::event() → case InputMethodQuery
→ QLineEdit::inputMethodQuery(ImEnabled)
→ return isEnabled() && !isReadOnly(); // true!

setEchoMode() 不直接发 InputMethodQuery，但修改 hints 后触发平台重新查询——返回值漏掉了 attribute 检查。

QLineEdit::setEchoMode → IBus → InputMethodQuery

Part 1 · 源码对比

Qt5 · 行为正确
// qwidget.cpp — Qt 5.15
case QEvent::InputMethodQuery:
  if (testAttribute(Qt::WA_InputMethodEnabled)) {
    // attribute 为 true，才进入查询流程
    QInputMethodQueryEvent *query = static_cast<…>(event);
    Qt::InputMethodQueries queries = query->queries();
    for (uint i = 0; i < 32; ++i) {
      Qt::InputMethodQuery q = Qt::InputMethodQuery(int(queries & (1 << i)));
      if (q) {
        QVariant v = inputMethodQuery(q); // 调虚函数
        if (q == Qt::ImEnabled && !v.isValid() && isEnabled())
          v = QVariant(true); // fallback: 没实现就默认 true
        query->setValue(q, v);
      }
    }
    query->accept();
  }
  // WA_InputMethodEnabled=false → 整个查询被跳过 → 输入法拿不到 ImEnabled=true
  break;

WA_InputMethodEnabled=false 作为外层 gate，整段查询逻辑都不会执行。

case QEvent::InputMethodQuery: {
  QInputMethodQueryEvent *query = static_cast<…>(event);
  Qt::InputMethodQueries queries = query->queries();
  for (uint i = 0; i < 32; ++i) {
    Qt::InputMethodQuery q = Qt::InputMethodQuery(int(queries & (1 << i)));
    if (q) {

      if (q == Qt::ImEnabled && !v.isValid() && isEnabled())
        v = QVariant(testAttribute(Qt::WA_InputMethodEnabled));
        // ↑ 只有虚函数返回 invalid 才回退读 attribute
      query->setValue(q, v);
    }
  }
  query->accept();
  break;
}
case Qt::ImEnabled:
  return isEnabled() && !isReadOnly();
  // ⚠️ 返回了有效 QVariant → fallback 不可达
  // ⚠️ 没有检查 testAttribute(Qt::WA_InputMethodEnabled)！

Part 1 · 对比总结
10 / 23
对比

Qt5 正常工作，Qt6 漏掉了属性检查

Qt5
没有显式 ImEnabled 分支，fallback 正常工作。QLineEdit 走 default → QWidget::inputMethodQuery() → 返回 invalid QVariant → QWidget::event() 的 fallback 读到 WA_InputMethodEnabled ✅
Qt6
新增了显式 ImEnabled 分支但漏掉了属性检查，且该分支返回有效值，导致 QWidget::event() 的 fallback 永远不会触发 ❌

同样的问题也存在于 QTextEdit 和 QPlainTextEdit。

三个文本控件都存在同类遗漏

前置：setEchoMode 设置了哪些 hints

IBus 声明不支持 HiddenTextCapability

// qlineedit.cpp — setEchoMode(Password) 会设置：
// ImhHiddenText | ImhSensitiveData | ImhNoAutoUppercase | ImhNoPredictiveText
// setInputMethodHints(imHints) → QInputMethod::update(Qt::ImHints)
// qibusplatforminputcontext.cpp
bool QIBusPlatformInputContext::hasCapability(Capability c) const {
  case HiddenTextCapability:
    return false; // QTBUG-40691 — 密码框不应弹出输入法
}
// qinputmethod.cpp
static inline bool platformSupportsHiddenText() {
  return inputContext && inputContext->hasCapability(HiddenTextCapability);
}
// → Linux + IBus: false
// qinputmethod.cpp — objectAcceptsInputMethod()
bool QInputMethodPrivate::objectAcceptsInputMethod(QObject *obj) {
    static const bool supportsHiddenText = platformSupportsHiddenText();
    QInputMethodQueryEvent query(
      supportsHiddenText ? Qt::ImEnabled : Qt::ImEnabled | Qt::ImHints);
    QGuiApplication::sendEvent(obj, &query);
    enabled = query.value(Qt::ImEnabled).toBool();
    if (enabled && !supportsHiddenText && hints.testFlag(ImhHiddenText))
      enabled = false; // ← 密码框：直接禁用
}

Part 1 · 根因
13 / 23
回到根因

密文正常 ≠ attribute 生效了

setEchoMode() 改的是 InputMethodHints（密码提示），不是 WA_InputMethodEnabled。

密文正常是因为 ImhHiddenText 让平台输入法主动退出（见上方 objectAcceptsInputMethod()），不等于 attribute 生效了。

切回 Normal 清除 hints 后，ImEnabled 查询走的是 QLineEdit 自己的分支——而这个分支漏掉了 attribute 检查。
两条禁用路径：ImhHiddenText（IBus 安全） vs WA_InputMethodEnabled（被绕过）

Part 1 · Review
15 / 23
论证策略

Reviewer的观点

Volker Hilsheimer
WA_InputMethodEnabled 是给 widget 作者用的，暴露给应用开发者合适吗？
Richard
依赖这个属性做密码安全很脆弱
Axel Spoerl
可访问性场景需要输入法；密码安全不止输入法层面
如果从"密码安全"角度论证：会引发一连串无法在 patch 里回答的问题——残障用户、安全 API 边界、内存层面处理……
最终论证方式：这是 ImEnabled 查询的一致性修复，不是密码安全策略变更。
四个论点：① 一致性：fallback 已有属性检查，三个 widget 的 override 让它成死代码，加上 testAttribute() 只是对齐设计意图 ② 默认行为不变：WA_InputMethodEnabled 默认 true ③ 恢复 Qt 5 行为：Qt 5.3 引入显式 ImEnabled 时属性检查被遗漏；WA_InputMethodEnabled 并不是废弃属性，这里是 Qt6 的回归 ④ 不声称是安全修复：安全场景由 ImhSensitiveData 处理
Gerrit Review · WA_InputMethodEnabled 不是废弃属性，这里是 Qt6 的回归
